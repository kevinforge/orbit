# Orbit 小白快速上手

这份指南面向第一次使用 Orbit 的初级程序员和小白用户。你不需要先理解多数字员工协作的内部原理，只要跟着步骤做，就可以让 Orbit 在你的本地项目里帮你拆需求、写代码、跑测试和总结结果。

## 你会用 Orbit 做什么

Orbit 是一个本地优先的数字员工工作台。你把一个本地项目目录交给 Orbit，然后在聊天框里把任务分配给不同数字员工，例如：

- `@architect:` 让架构师先看代码、拆方案、评估风险。
- `@developer:` 让开发人员修改代码、运行命令、提交变更。
- `@tester:` 让测试人员验证功能、跑测试、报告问题。
- `@supervisor:` 让监督员在会话里帮助推进任务闭环。
- `@all:` 同时把任务发给当前启用的多个数字员工。

如果你还不熟悉这些角色，可以先使用推荐的“多数字员工协作”模板，让 Orbit 自动启用适合协作开发的默认配置。

## 第 0 步：准备环境

先安装 Node.js。npm 会随 Node.js 一起安装，不需要单独安装。

Windows 新手推荐这样做：

1. 打开 Node.js 官网：https://nodejs.org/
2. 下载 LTS 版本，版本号需要是 20 或更高。
3. 双击安装包，一路按默认选项安装即可。
4. 安装完成后，关闭当前 PowerShell，再重新打开一个新的 PowerShell。

如果你使用 macOS 或 Linux，也可以从 Node.js 官网下载对应安装包；已经熟悉命令行的用户也可以使用 `nvm`、`fnm` 或系统包管理器安装 Node.js 20+。

然后确认电脑里已经有 Node.js 和 npm：

```powershell
node --version
npm --version
```

如果 `node --version` 命令不存在，或者版本低于 20，请重新安装新版 Node.js。如果 `node` 正常但 `npm` 不存在，通常也是 Node.js 安装不完整，建议重新安装 LTS 版本。

Orbit 本身负责提供网页工作台；真正执行任务的是背后的命令行工具。你至少需要安装一个数字员工运行时，建议新手优先安装你最常用、已经登录好的那个：

| 运行时 | 用途 | 安装方式 |
| --- | --- | --- |
| Claude Code | 常用于开发人员、监督员 | `npm install -g @anthropic-ai/claude-code` |
| OpenAI Codex | 常用于产品、架构、通用任务 | `npm install -g @openai/codex` |
| CodeBuddy | 常用于测试或自定义员工 | 参考 CodeBuddy 官方 CLI 文档 |

安装后，请在终端里单独运行对应 CLI，并按它自己的提示完成登录或授权。Orbit 只能调用已经能在你本机正常运行的 CLI。

## 第 1 步：安装 Orbit

打开 PowerShell，执行：

```powershell
npm install -g orbit
```

安装完成后，启动 Orbit：

```powershell
orbit
```

看到服务启动后，在浏览器打开：

```text
http://localhost:4317
```


然后打开 `http://localhost:4320`。

## 第 2 步：创建第一个工作区

工作区就是你要让数字员工处理的本地项目目录。

1. 打开 Orbit 页面后，点击左侧“工作区”旁边的 `+`。
2. 选择一个本地项目目录，例如你的前端、后端或脚本项目。
3. 如果出现模板选择，第一次使用建议选择“多数字员工协作”。
4. 创建成功后，左侧会出现该工作区和一个新会话。

选择“多数字员工协作”时，Orbit 会尽量启用架构师、开发人员、测试人员和监督员，并把它们配置到当前可用的运行时上。这样新手不需要一开始就手动调很多权限和提示词。

如果你选择“空白工作区”，默认数字员工模板不会自动启用。你需要继续完成下一步。

## 第 3 步：启用数字员工

如果左侧“数字员工”区域提示还没有启用数字员工：

1. 点击“数字员工”旁边的 `+`。
2. 在弹窗里打开需要的默认模板开关。
3. 新手建议至少启用 `developer`；如果想要更稳妥的协作流程，再启用 `architect`、`tester` 和 `supervisor`。
4. 检查每个数字员工的“运行时”是否显示可用。
5. 点击“保存”。

如果 Orbit 提示某个运行时未安装，先按页面上的安装入口安装对应 CLI。安装完成后，回到 Orbit 点击“重新检测运行环境”。

## 第 4 步：发出第一条任务

在底部输入框里输入一条带 `@数字员工:` 的任务，然后按 Enter 发送。

例如，让架构师先看项目：

```text
@architect: 请先阅读这个项目的结构，用小白能懂的话说明它是做什么的，并告诉我如果要新增一个功能应该从哪里开始。
```

如果你想直接让开发人员动手，可以写得更具体：

```text
@developer: 请帮我在 README 里补充本地启动步骤。改动要尽量小，完成后运行相关检查，并告诉我改了哪些文件。
```

如果你启用了监督员，也可以直接描述目标，让监督员帮助分配：

```text
请帮我给这个项目增加一个登录页的表单校验，要求先评估方案，再实现，再测试。
```

发送后，你会看到对应数字员工进入运行状态。它可能会读取文件、运行命令、写代码或把任务继续交给其他数字员工。

## 第 5 步：看懂运行结果

数字员工回复时，请重点看三类信息：

- 它理解的目标是否正确。
- 它实际改了哪些文件、运行了哪些检查。
- 它是否说明还有失败、限制或需要你确认的地方。

如果任务还在排队，Orbit 会按每个数字员工自己的队列顺序执行。如果你发现方向不对，可以点击输入框旁边的“打断”，停止后续自动协作，然后重新说明你的需求。

## 第 6 步：推荐的新手工作流

第一次使用时，可以按下面的节奏来：

1. 先问 `@architect:`：“这个需求应该怎么做？风险是什么？”
2. 方案满意后，让 `@developer:`：“按刚才方案实现，改动要小，完成后运行测试。”
3. 开发完成后，让 `@architect:`：“请 review 刚才的改动，指出问题。”
4. 没有明显问题后，让 `@tester:`：“请运行验证，并说明是否可以交付。”
5. 最后让数字员工总结：“请用三句话告诉我改了什么、怎么验证、还有什么风险。”

这比一上来就要求“直接全自动完成”更适合新手，因为你能在每一步看懂发生了什么，也更容易及时纠偏。

## 常见问题

### 页面提示运行时未安装

说明 Orbit 没检测到对应 CLI。请先在终端里确认命令能单独运行：

```powershell
claude --version
codex --version
codebuddy --version
```

只需要检查你实际使用的运行时即可。安装或登录完成后，回到 Orbit 点击“重新检测运行环境”。

### 输入框不能发送

通常是因为还没有创建工作区，或者当前工作区没有启用任何数字员工。先创建工作区，再到“数字员工”面板启用至少一个员工。

### 数字员工没有修改文件

先确认你指派的是有写入权限的数字员工。默认情况下，`developer` 适合写代码；`architect` 和 `tester` 更适合读代码、评审和验证。

### 我不想让数字员工改整个电脑

Orbit 的工作区指向你选择的本地项目目录。新手建议每次只选择一个明确的项目目录，不要选择系统目录、用户主目录或磁盘根目录。

### Orbit 会把数据放在哪里

Orbit 是本地优先的产品，会把工作区、会话、消息、运行记录等数据保存在你电脑的 `~/.orbit` 目录下。

## 下一步可以尝试

- 给同一个项目创建多个会话，分别处理不同需求。
- 在“数字员工”面板里复制一个默认模板，改成你自己的专用员工。
- 在工作区配置里使用内置模板，让所有会话都遵循同一套协作规则。
- 让数字员工先解释代码，再让它改代码。对小白用户来说，“先理解再修改”通常更稳。
